### 1. Понятие хранилищ данных, баз знаний

Хранилище данных — это такая большая, централизованная база данных, где собираются данные из разных источников. Ее основная задача – анализ и поддержка принятия решений, а не операционная работа. База знаний — это, по сути, структурированный набор информации, который включает факты, правила и логические выводы. Она используется в системах искусственного интеллекта для решения сложных задач.

### 3. Как работают подстановочные знаки в SQL
Подстановочные знаки в SQL используются вместе с оператором `LIKE` для поиска данных, которые соответствуют определенному шаблону.
* `%` (процент) — заменяет любое количество символов (включая ноль). Например, `SELECT * FROM Products WHERE Name LIKE 'А%';` найдет все продукты, названия которых начинаются на "А".
* `_` (нижнее подчеркивание) — заменяет ровно один символ. Например, `SELECT * FROM Users WHERE Login LIKE 'user_';` найдет "user1", "user2" и так далее.

### 3. Какие типы СУБД в соответствии с моделями данных вы знаете
По моделям данных СУБД делятся на несколько типов:
* Реляционные СУБД (РСУБД) — самые распространенные. Данные хранятся в таблицах со связями между ними. Примеры: MySQL, PostgreSQL, SQL Server.
* Иерархические СУБД — данные организованы в виде дерева, где есть родительские и дочерние узлы.
* Сетевые СУБД — более гибкие, чем иерархические, позволяют одному дочернему узлу иметь несколько родителей.
* Объектно-ориентированные СУБД — хранят данные как объекты, как в объектно-ориентированном программировании.
* NoSQL СУБД — это целое семейство, включающее документоориентированные, колоночные, графовые и Key-Value СУБД. Они хороши для больших объемов неструктурированных данных.

### 4. Что такое первичный ключ
Первичный ключ – это одно или несколько полей (столбцов) в таблице, которые однозначно идентифицируют каждую запись (строку). Он не может содержать повторяющиеся значения и не может быть `NULL`. Например, `ID` пользователя в таблице `Пользователи` будет первичным ключом.

### 5. Что такое внешний ключ
Внешний ключ — это поле или набор полей в одной таблице, которые ссылаются на первичный ключ в другой таблице. Он используется для установления связей между таблицами, обеспечивая целостность данных. Например, `ID_продукции` в таблице `Производство` может быть внешним ключом, ссылающимся на `ID` в таблице `Продукция`.

### 6. Основные принципы построения концептуальной модели данных
Концептуальная модель данных описывает суть данных и отношения между ними с точки зрения пользователя, без привязки к конкретной СУБД. Основные принципы:
* Высокий уровень абстракции: фокусируется на "что", а не на "как".
* Независимость от реализации: не зависит от технологий.
* Представление сущностей и связей: использует сущности (объекты реального мира) и связи между ними.
* Атрибуты: описывает характеристики сущностей.
* Понимание предметной области: создается на основе глубокого понимания требований бизнеса.

### 7. Основные принципы построения логической модели данных
Логическая модель данных — это представление данных, которое уже учитывает конкретную модель данных (например, реляционную), но еще не привязано к конкретной СУБД.
* Трансляция сущностей в таблицы: каждая сущность из концептуальной модели становится таблицей.
* Определение столбцов: атрибуты сущностей становятся столбцами таблиц.
* Определение первичных и внешних ключей: устанавливаются связи между таблицами.
* Нормализация: приведение таблиц к нормальным формам для уменьшения избыточности и улучшения целостности.
* Индексация: планирование индексов для оптимизации запросов.

### 8. Основные принципы построения физической модели данных
Физическая модель данных — это низкоуровневое описание базы данных, учитывающее конкретную СУБД и ее особенности.
* Определение типов данных: выбор конкретных типов данных для каждого столбца (например, `INT`, `VARCHAR`).
* Создание индексов: физическое создание индексов для ускорения поиска.
* Размещение данных: определение способов хранения данных на диске (например, файловые группы).
* Партиционирование: разбиение больших таблиц на более мелкие части для улучшения производительности.
* Физическое хранение ключей: реализация первичных и внешних ключей в выбранной СУБД.

### 9. Основные принципы структуризации и нормализации базы данных.
Структуризация данных — это организация данных таким образом, чтобы они были логически связаны и легко доступны.
Нормализация — это процесс организации данных в базе данных для уменьшения избыточности и улучшения целостности данных. Это достигается путем разделения больших таблиц на более мелкие и установления связей между ними. Есть разные нормальные формы (1NF, 2NF, 3NF и т.д.), каждая из которых устраняет определенные виды аномалий.

### 10. Методы организации целостности данных
Целостность данных — это обеспечение точности, полноты и непротиворечивости данных.
* Целостность сущностей: каждая строка в таблице должна быть уникально идентифицирована (через первичный ключ).
* Ссылочная целостность: все внешние ключи должны ссылаться на существующие первичные ключи или быть `NULL`.
* Целостность доменов: значения в столбцах должны соответствовать предопределенным типам данных и ограничениям (например, `CHECK` ограничения).
* Пользовательская целостность: любые другие правила, которые не подпадают под вышеперечисленные, например, через триггеры или хранимые процедуры.

### 11. Способы контроля доступа к данным
Контроль доступа к данным — это ограничение того, кто может просматривать, изменять или удалять данные.
* Управление доступом на основе ролей (RBAC): пользователям назначаются роли, а ролям — разрешения.
* Дискреционный контроль доступа (DAC): владелец объекта данных сам определяет, кто имеет к нему доступ.
* Мандатный контроль доступа (MAC): доступ определяется уровнем конфиденциальности данных и уровнем допуска пользователя.
* Аутентификация: проверка личности пользователя (логин/пароль).
* Авторизация: определение прав пользователя после аутентификации.

### 12. Способы управления привилегиями
Управление привилегиями (правами доступа) в СУБД осуществляется через команды `GRANT` и `REVOKE`.
* `GRANT` — выдача привилегий. Например, `GRANT SELECT ON Products TO EmployeeRole;` дает роль "EmployeeRole" право на чтение данных из таблицы "Products".
* `REVOKE` — отмена привилегий. Например, `REVOKE UPDATE ON Production FROM EmployeeRole;` отменяет право на обновление данных в таблице "Production" для "EmployeeRole".
Привилегии могут быть на уровне сервера, базы данных, схемы, таблицы, столбца или хранимой процедуры.

### 13. Реляционное исчисление
Реляционное исчисление — это теоретическая основа реляционных баз данных. Оно описывает, как получать данные из реляционных баз данных, используя математическую логику, а не пошаговые инструкции. Есть два основных типа:
* Кортежное реляционное исчисление: формулирует запросы как предикаты, которые описывают свойства кортежей (строк), которые должны быть включены в результат.
* Доменное реляционное исчисление: формулирует запросы, описывая свойства значений в доменах (столбцах).
Это важно для понимания того, как работают языки запросов вроде SQL.

### 14. Компоненты и структуры данных
Компоненты СУБД включают:
* Ядро СУБД: отвечает за хранение, извлечение и управление данными.
* Процессор запросов: анализирует и оптимизирует SQL-запросы.
* Менеджер транзакций: обеспечивает атомарность, согласованность, изоляцию и долговечность транзакций.
* Менеджер буферов: управляет данными в оперативной памяти.
* Системный каталог (словарь данных): хранит метаданные о базе данных (структура таблиц, индексов, пользователей).
Структуры данных в СУБД:
* Таблицы: основные единицы хранения данных.
* Индексы: структуры для ускорения поиска данных.
* Представления (VIEWs): виртуальные таблицы, основанные на запросах.
* Журналы транзакций: записи обо всех изменениях данных для восстановления.

### 15. Методы описания схем баз данных в современных СУБД
Схемы баз данных описывают структуру базы данных, включая таблицы, столбцы, связи, индексы и ограничения.
* Язык определения данных (DDL) SQL: основные команды `CREATE TABLE`, `ALTER TABLE`, `DROP TABLE` и т.д.
* Диаграммы сущность-связь (ER-диаграммы): графическое представление концептуальной модели.
* Инструменты CASE (Computer-Aided Software Engineering): специализированное ПО для проектирования баз данных (например, ERWin, PowerDesigner).
* XML Schema Definition (XSD): для описания структуры XML-документов, которые могут использоваться для обмена данными.

### 16. Структуры данных СУБД
Ключевые структуры данных в СУБД включают:
* Таблицы (отношения): фундаментальные структуры, состоящие из строк (кортежей) и столбцов (атрибутов).
* Индексы: используются для быстрого поиска и извлечения данных. Могут быть B-деревья, хеш-таблицы и т.д.
* Представления (VIEWs): виртуальные таблицы, которые не хранят данные, но предоставляют определенный "взгляд" на них.
* Кластеры: способ физического хранения связанных данных вместе для ускорения доступа.
* Журналы (логи): используются для отслеживания изменений и восстановления данных после сбоев.

### 17. Общий подход к организации представлений, таблиц, индексов и кластеров
* Таблицы: основа. Данные хранятся в строках и столбцах. При проектировании таблиц важно соблюдать нормальные формы.
* Представления (VIEWs): это как сохраненный запрос. Они упрощают сложные запросы, обеспечивают безопасность (показывают только нужные данные) и совместимость.
* Индексы: создаются на одном или нескольких столбцах таблицы для ускорения операций поиска и сортировки. Без индексов СУБД пришлось бы сканировать всю таблицу.
* Кластеры: это физический способ хранения данных. Связанные строки из одной или нескольких таблиц хранятся вместе на диске, что ускоряет запросы, которые часто объединяют эти данные.

### 18. Способы контроля доступа к данным
Это уже было, но повторюсь, что нужно для экзамена! Контроль доступа к данным — это механизмы, которые определяют, кто и как может взаимодействовать с данными.
* Аутентификация: проверка личности пользователя (логин и пароль, сертификаты).
* Авторизация: предоставление или ограничение прав доступа к конкретным объектам базы данных (таблицам, представлениям, процедурам).
* Роли: группы привилегий, которые назначаются пользователям.
* Шифрование: защита данных от несанкционированного доступа при хранении и передаче.
* Аудит: запись действий пользователей для отслеживания возможных нарушений.

### 19. Управление привилегиями
Опять повторяется, но повторение - мать учения! Управление привилегиями — это процесс выдачи и отзыва прав доступа пользователям или ролям к ресурсам базы данных.
* `GRANT`: оператор SQL для выдачи привилегий. Например, `GRANT SELECT, INSERT ON Продукция TO СотрудникПроизводства;`.
* `REVOKE`: оператор SQL для отзыва привилегий. Например, `REVOKE DELETE ON Продукция FROM СотрудникПроизводства;`.
Привилегии могут быть на системном уровне (например, создавать базы данных) или на объектном уровне (например, читать из таблицы).

### 20. Алгоритм проведения процедуры резервного копирования.
Резервное копирование — это создание копии данных для их восстановления в случае сбоя. Общий алгоритм:
### 1. Определить стратегию: полное, дифференциальное или инкрементное копирование.
### 2. Выбрать время: когда нагрузка на СУБД минимальна.
### 3. Использовать инструменты СУБД: например, `BACKUP DATABASE` в SQL Server.
### 4. Проверить целостность бэкапа: убедиться, что бэкап не поврежден.
### 5. Сохранить бэкап: на отдельном носителе, желательно в удаленном месте.
### 6. Вести журнал: записывать информацию о каждом бэкапе.
### 7. Периодически тестировать восстановление: это очень важно, чтобы убедиться, что бэкапы работают.

### 21. Модели восстановления SQL-сервера
В SQL Server есть три основные модели восстановления, которые определяют, как СУБД управляет журналом транзакций и как можно восстанавливать данные:
* Простая (Simple): журнал транзакций автоматически усекается после создания контрольных точек. Можно восстановить базу данных только до последнего полного или дифференциального бэкапа.
* Полная (Full): все транзакции записываются в журнал. Позволяет восстанавливать базу данных до любой точки во времени (point-in-time recovery). Требует регулярного резервного копирования журнала транзакций.
* С неполным протоколированием (Bulk-Logged): похожа на полную, но для массовых операций (например, импорт больших объемов данных) изменения записываются минимально, что уменьшает размер журнала, но ограничивает возможности восстановления до точки во времени для этих операций.

### 22. Резервное копирование баз данных
Резервное копирование — это процесс создания копий данных базы данных. Это критически важно для защиты от потери данных из-за сбоев оборудования, ошибок программного обеспечения, человеческого фактора или вредоносных атак.
Виды резервных копий:
* Полная: копирует всю базу данных.
* Дифференциальная: копирует только те данные, которые изменились с момента последнего *полного* бэкапа.
* Инкрементная: копирует только те данные, которые изменились с момента последнего *любого* бэкапа (полного, дифференциального или инкрементного).

### 23. Восстановление баз данных
Восстановление баз данных — это процесс возврата базы данных в согласованное состояние после сбоя с использованием резервных копий и журналов транзакций.
Основные шаги:
### 1. Восстановление полного бэкапа.
### 2. Восстановление последнего дифференциального бэкапа (если есть).
### 3. Применение бэкапов журнала транзакций (если используется полная модель восстановления) для восстановления до нужной точки во времени.
Важно регулярно тестировать процесс восстановления, чтобы убедиться в его работоспособности.

### 24. Аутентификация пользователей в СУБД
Аутентификация — это процесс проверки личности пользователя, который пытается получить доступ к СУБД.
Распространенные методы:
* Аутентификация по логину и паролю: самый распространенный метод.
* Аутентификация Windows (интегрированная аутентификация): СУБД доверяет аутентификации, выполненной операционной системой.
* Сертификаты: использование цифровых сертификатов для подтверждения личности.
* Многофакторная аутентификация (MFA): использование нескольких методов для усиления безопасности.

### 25. Назначение серверных ролей и ролей баз данных
Роли в СУБД — это наборы разрешений, которые можно назначить пользователям, упрощая управление доступом.
* Серверные роли: определяют разрешения на уровне экземпляра SQL Server. Например, роль `sysadmin` имеет полные административные права на весь сервер. `dbcreator` может создавать базы данных.
* Роли баз данных: определяют разрешения на уровне конкретной базы данных. Например, `db_datareader` может читать данные из всех таблиц в базе, а `db_datawriter` может записывать данные.
Пользователи могут быть членами одной или нескольких ролей, получая совокупность их привилегий.

### 26. Авторизация пользователей при получении доступа к ресурсам
Авторизация — это процесс определения того, какие действия разрешены аутентифицированному пользователю в СУБД.
После того как пользователь аутентифицирован (его личность подтверждена), система авторизации проверяет его права на выполнение конкретных операций с конкретными объектами.
Примеры: пользователю разрешено `SELECT` из таблицы `Products`, но запрещено `DELETE` из нее. Авторизация часто реализуется через роли и индивидуальные привилегии.

### 27. Домены в СУБД
В контексте баз данных, домен — это набор допустимых значений для определенного атрибута (столбца).
Например, для столбца "Возраст" домен может быть "целые числа от 0 до 150". Для столбца "Пол" домен может быть "Мужской", "Женский".
Домены помогают обеспечить целостность данных, гарантируя, что значения в столбцах соответствуют определенным правилам и ограничениям.

### 28. Триггеры: понятие, назначение, синтаксис
Триггер — это специальный тип хранимой процедуры, которая автоматически выполняется в ответ на определенные события в базе данных (например, `INSERT`, `UPDATE` или `DELETE` на таблице).
Назначение:
* Обеспечение сложной бизнес-логики и целостности данных.
* Ведение аудита изменений.
* Автоматическое обновление связанных данных.
Синтаксис (обобщенный):

```sql
CREATE TRIGGER Имя_Триггера
ON Имя_Таблицы
FOR | AFTER | INSTEAD OF [INSERT, UPDATE, DELETE]
AS
BEGIN
-- SQL-код, который будет выполнен
END;
```

### 29. Транзакции: понятие, назначение, синтаксис
Транзакция — это логическая единица работы, состоящая из одного или нескольких операторов SQL, которые выполняются как единое целое. Либо все операции в транзакции завершаются успешно (фиксируются), либо ни одна из них не применяется (откатываются).
Назначение:

* Обеспечение целостности данных, особенно при многопользовательском доступе.
* Соблюдение свойств ACID: Атомарность, Согласованность, Изоляция, Долговечность.
Синтаксис:

```sql
BEGIN TRANSACTION; -- Начало транзакции
-- SQL-операторы
COMMIT; -- Фиксация (сохранение) изменений
ROLLBACK; -- Откат (отмена) изменений
```

### 30. Индексы: понятие, назначение, синтаксис
Индекс — это структура данных, которая улучшает скорость операций поиска данных в таблице базы данных.
Назначение:

* Ускорение запросов `SELECT`.
* Ускорение операций `JOIN` и `ORDER BY`.
* Обеспечение уникальности значений (для уникальных индексов).
Синтаксис:
```sql
CREATE [UNIQUE] INDEX Имя_Индекса
ON Имя_Таблицы (Столбец1 [ASC|DESC], Столбец2 [ASC|DESC], ...);
```
Например: `CREATE INDEX IX_Production_Date ON Production (Дата);`

### 31. SQL-роли: понятие, назначение, синтаксис
SQL-роль — это набор привилегий (прав доступа), который создается и управляется как единое целое, а затем назначается пользователям.
Назначение:

* Упрощение управления разрешениями: вместо назначения прав каждому пользователю, достаточно назначить их роли, а затем назначить роль пользователям.
* Повышение безопасности: снижается вероятность ошибок при управлении доступом.
Синтаксис:

```sql
CREATE ROLE Имя_Роли;
GRANT SELECT ON Products TO Имя_Роли;
ADD MEMBER Имя_Пользователя TO Имя_Роли;
```

### 32. Хранимые процедуры: понятие, назначение, синтаксис
Хранимая процедура — это набор SQL-операторов, скомпилированных и хранящихся в базе данных.
Назначение:

* Повышение производительности: скомпилированный код выполняется быстрее.
* Повторное использование кода: один раз написанная процедура может быть вызвана многократно.
* Безопасность: пользователи могут выполнять процедуры, не имея прямых прав на базовые таблицы.
* Инкапсуляция бизнес-логики.
Синтаксис (обобщенный):

```sql
CREATE PROCEDURE Имя_Процедуры
@Параметр1 ТипДанных,
@Параметр2 ТипДанных
AS
BEGIN
-- SQL-операторы
END;
```

### 33. Привилегии доступа и передача привилегий
Привилегии доступа — это конкретные разрешения, которые позволяют пользователю или роли выполнять определенные действия с объектами базы данных (например, `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `EXECUTE`).
Передача привилегий означает возможность пользователя, которому даны определенные привилегии, передать их другим пользователям. Это делается с помощью опции `WITH GRANT OPTION` в команде `GRANT`.
Пример: `GRANT SELECT ON Products TO UserA WITH GRANT OPTION;` позволит `UserA` не только читать данные из `Products`, но и давать право `SELECT` другим.

### 34. Правила синтаксиса SQL, основные типы данных
Правила синтаксиса SQL:

* Нечувствительность к регистру (обычно, но не всегда).
* Завершение операторов точкой с запятой (рекомендуется).
* Ключевые слова не сокращаются.
* Комментарии: `-- однострочный` или `/* многострочный */`.
* Пробелы и переводы строк игнорируются (в разумных пределах).
Основные типы данных:
* Числовые: `INT` (целые числа), `DECIMAL`/`NUMERIC` (точные десятичные), `FLOAT`/`REAL` (числа с плавающей точкой).
* Строковые: `VARCHAR` (переменная длина), `CHAR` (фиксированная длина), `TEXT` (большие строки).
* Дата/Время: `DATE`, `TIME`, `DATETIME`, `TIMESTAMP`.
* Булевы: `BOOLEAN` (True/False).
* Двоичные: `VARBINARY`, `BLOB`.

### 35. Обеспечение безопасности служб AD DS
AD DS (Active Directory Domain Services) — это служба каталогов Microsoft, которая хранит информацию о сетевых ресурсах и пользователях. Безопасность AD DS важна для общей безопасности инфраструктуры.

* Использование сложных паролей и политик блокировки учетных записей.
* Принцип наименьших привилегий: назначать пользователям и службам только те права, которые им необходимы.
* Регулярное обновление системы и применение патчей безопасности.
* Мониторинг событий безопасности и аудит доступа.
* Защита контроллеров домена: физическая безопасность, ограничение доступа.
* Сегментация сети: изоляция критически важных компонентов.

### 36. Мониторинг, управление и восстановление AD DS

* Мониторинг:
* Использование "Просмотра событий" (Event Viewer) для отслеживания ошибок и предупреждений.
* Инструменты мониторинга производительности (Performance Monitor) для отслеживания загрузки контроллеров домена.
* Специализированные инструменты мониторинга AD DS (например, Azure AD Connect Health).
* Управление:
* Использование консолей "Пользователи и компьютеры Active Directory", "Сайты и службы Active Directory", "Домены и доверие Active Directory".
* Групповые политики для централизованного управления конфигурациями.
* PowerShell для автоматизации задач.
* Восстановление:
* Регулярное резервное копирование состояния системы контроллеров домена.
* Авторитетное и неавторитетное восстановление: при восстановлении одного контроллера домена или всей службы.
* Использование журнала DSRM (Directory Services Restore Mode) для восстановления контроллеров домена.

### 37. Внедрение групповых политик
Групповые политики (Group Policies, GP) — это инфраструктура в Windows для централизованного управления настройками пользователей и компьютеров в домене Active Directory.

* Создание GPO (Group Policy Object): с помощью консоли "Управление групповой политикой".
* Настройка параметров: в GPO можно настроить параметры безопасности, установки программного обеспечения, сценарии запуска/завершения и многое другое.
* Связывание GPO: GPO связываются с сайтами, доменами или организационными единицами (OU), чтобы применять политики к соответствующим объектам.
* Фильтрация: можно использовать фильтры безопасности или WMI-фильтры для применения политики только к определенным пользователям или компьютерам.

### 38. Управление параметрами пользователей с помощью групповых политик
Групповые политики позволяют гибко управлять настройками пользовательской среды:

* Настройки безопасности: требования к сложности паролей, блокировка учетных записей, права пользователей.
* Настройки рабочего стола: фон, заставки, доступ к панели управления.
* Распределение программного обеспечения: автоматическая установка приложений на компьютеры пользователей.
* Перенаправление папок: папки пользователя (например, "Мои документы") могут храниться на сетевом ресурсе.
* Настройки сети и интернет-безопасности.
* Скрипты входа/выхода из системы.

### 39. Обеспечение безопасного доступа к общим файлам
Безопасный доступ к общим файлам важен для защиты конфиденциальной информации.

* Использование разрешений NTFS: настройка прав доступа на уровне файлов и папок (чтение, запись, изменение, полный доступ) для пользователей и групп.
* Разрешения общего доступа (Share Permissions): определяют, кто может подключаться к сетевой папке. Рекомендуется использовать разрешения NTFS для более детального контроля.
* Принцип наименьших привилегий: предоставлять только минимально необходимые права.
* Аудит доступа к файлам: отслеживание попыток доступа к файлам для выявления подозрительной активности.
* Шифрование: использование EFS (Encrypting File System) или BitLocker для защиты файлов.
* Резервное копирование: регулярное создание копий общих ресурсов.

### 40. Установка и настройка SQL-сервера
Установка SQL Server включает несколько шагов:
#### 1. Планирование: выбор редакции, компонентов, места установки.
#### 2. Запуск установочного мастера: выбор типа установки (новая установка, добавление компонентов).
#### 3. Выбор компонентов: Database Engine, Reporting Services, Analysis Services и др.
#### 4. Настройка экземпляра: именованный или по умолчанию.
#### 5. Конфигурация служб: учетные записи для служб SQL Server (например, Agent, Database Engine).
#### 6. Режим проверки подлинности: Windows-аутентификация, смешанный режим.
#### 7. Настройка параметров баз данных: пути к файлам данных и журналов.
После установки:

* Обновление до последних пакетов обновлений.
* Настройка памяти и процессора.
* Настройка безопасности (логины, роли).
* Настройка сетевых протоколов.

### 41. Импорт и экспорт данных
Импорт и экспорт данных — это процессы перемещения данных между SQL Server и другими источниками/приемниками.

* Инструменты SQL Server:
* SQL Server Integration Services (SSIS): мощный инструмент для создания сложных ETL (Extract, Transform, Load) процессов.
* Мастер импорта и экспорта SQL Server: простой графический интерфейс для выполнения базовых задач.
* Команды `BULK INSERT` / `BCP` (Bulk Copy Program): для высокопроизводительного импорта/экспорта больших объемов данных из/в текстовые файлы.
* Форматы данных: CSV, Excel, XML, текстовые файлы, другие базы данных.
* Назначение: миграция данных, интеграция с другими системами, создание отчетов.

### 42. Выполнение мониторинга SQL Server с использованием оповещений и предупреждений
Мониторинг SQL Server важен для обеспечения производительности, доступности и безопасности.

* Системные хранимые процедуры и функции: например, `sp_who`, `sys.dm_os_wait_stats`.
* Динамические административные представления (DMV): предоставляют информацию о состоянии сервера и базы данных в реальном времени.
* SQL Server Profiler / Extended Events: для трассировки и анализа событий на сервере.
* Монитор производительности (Performance Monitor, PerfMon): для отслеживания системных счетчиков.
* Агенты SQL Server и оповещения: можно настроить задания и оповещения для уведомления о проблемах (например, заполнение диска, блокировки, ошибки).
* Средства сторонних разработчиков: специализированные инструменты для мониторинга SQL Server.

### 43. Современные инструментальные средства проектирования схемы базы данных
Современные инструментальные средства значительно упрощают проектирование и управление базами данных.

* CASE-средства (Computer-Aided Software Engineering):
* ER/Studio, PowerDesigner, DBeaver, MySQL Workbench.
* Позволяют создавать ER-диаграммы, генерировать DDL-скрипты, реверс-инжиниринг существующих баз данных.
* Интегрированные среды разработки (IDE) СУБД:
* SQL Server Management Studio (SSMS), DataGrip, Visual Studio с SQL Server Data Tools.
* Предоставляют редакторы кода, инструменты для управления объектами базы данных, отладки запросов.
* Онлайн-инструменты и облачные сервисы:
* dbdiagram.io, draw.io (с плагинами для ERD).
* Облачные платформы баз данных (Azure SQL Database, Amazon RDS) с веб-интерфейсами для управления.
